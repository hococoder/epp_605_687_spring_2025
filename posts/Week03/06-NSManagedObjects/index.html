Week 3 - Making NSManagedObjects

Previous: Setting up a Schema

Next: Integrating Core Data



Overview

There has to be something behind the Schema we discussed in the last module - each entity has to be connected to an underlying class so that it can be compiled and instantiated like any other type in our code. Let's look at a few different ways to make these objects.

Declaring NSManagedObjects Manually

In your data model, each entity is represented under the hood by a specific class, which is a child of NSManagedObject. You don’t have to create these classes explicitly - one of the main features of Core Data - but you can if you want to add extra functionality to those classes. All you have to do is choose an entity and select the "Editor -> Create NSManagedObject subclass" menu in Xcode. Xcode will then make a class with a body that looks like the one below. You can then customize this class - but keep in mind if you update the schema, you have to keep this class up to date as well.

import CoreData

class DBSample: NSManagedObject {
  
  @NSManaged var name: String
  @NSManaged var amount: String
  
}

Keeping track of these classes can be a pain; luckily Core Data and Xcode have built in functionality that does this for us.

Letting Xcode Do the Hard Work

Let's take a look at how CoreData creates class types from the entities you setup in your model, and how you can customize those classes to add extra functionality. If we go over to ContentView.swift and make an instance of TeacherEntity with

let teacherEntity = TeacherEntity()

It compiles! Go ahead and remove it from ContentView.swift so you don't forget later.

But we don’t have a TeachEntity class defined in our project… or do we? Let’s go over to the TeacherEntity in the data model editor. You can see in the properties pane that there is a section called Codegen, and currently selected is Class Definition. What does that mean? Well, when Xcode builds a project, it uses space in the DerivedData folder as a scratch space, building a series of intermediate, derived, files in the process, an in our particular case, ones that are generated from the CoreData model. Let’s open a console window and go to the following folder under your home directory:

Library/Developer/Xcode/DerivedData

Doing a file listing, I see the different apps I have derived data for, and I can go to our EPGradeBook app. Navigate down to the following folder:

Build/Intermediates.noindex/EPGradeBook.build/Debug-iphonesimulator/EPGradeBook.build/DerivedSources/CoreDataGenerated/EPGradeBook

Within that folder, you can see a series of Swift files that were automatically generated by the build system, based on the information in the core data model. Using the more command on the AssignmentEntity+CoreDataClass.swift file in the terminal, and you can see that the AssignmentEntity class declaration is here, and it extends the NSManagedObject class.

//
//  AssignmentEntity+CoreDataClass.swift
//
//
//  Created by Teacher on 2/6/21.
//
//  This file was automatically generated and should not be edited.
//

import Foundation
import CoreData

@objc(AssignmentEntity)
public class AssignmentEntity: NSManagedObject {

}

The other file - AssignmentEntity+CoreDataProperties.swift is a little more exciting. It has the attributes we assigned - even an NSSet to represent the grades collection - along with some helper methods to fetch all of the AssignmentEntity entries, as well as add grades to or delete grades from the assignment.

//
//  AssignmentEntity+CoreDataProperties.swift
//
//
//  Created by Teacher on 2/6/21.
//
//  This file was automatically generated and should not be edited.
//

import Foundation
import CoreData


extension AssignmentEntity {

    @nonobjc public class func fetchRequest() -> NSFetchRequest<AssignmentEntity> {
        return NSFetchRequest<AssignmentEntity>(entityName: "AssignmentEntity")
    }

    @NSManaged public var assignmentDescription: String?
    @NSManaged public var assignmentType: String?
    @NSManaged public var averageRating: Double
    @NSManaged public var dueDate: Date?
    @NSManaged public var id: UUID?
    @NSManaged public var maxGrade: Double
    @NSManaged public var name: String?
    @NSManaged public var releaseDate: Date?
    @NSManaged public var grades: NSSet?

}

// MARK: Generated accessors for grades
extension AssignmentEntity {

    @objc(addGradesObject:)
    @NSManaged public func addToGrades(_ value: AssignmentGradeEntity)

    @objc(removeGradesObject:)
    @NSManaged public func removeFromGrades(_ value: AssignmentGradeEntity)

    @objc(addGrades:)
    @NSManaged public func addToGrades(_ values: NSSet)

    @objc(removeGrades:)
    @NSManaged public func removeFromGrades(_ values: NSSet)

}

extension AssignmentEntity : Identifiable {

}

You’ll notice both of these files have a comment saying that they are autogenerated and shouldn’t be edited - and indeed, these files are managed and re-derived by the build system when you make changes. So how do you customize the AssignmentEntity class if you want to add other functionality? Well, you can use the same technique that the AssignmentEntity+CoreDataProperties file used - an extension. Let’s make a new file called AssignmentEntity+Extension in the project, and in it, setup an extension for the AssignmentEntity class, and define a method to calculate the average of the assignment’s grades.

extension AssignmentEntity {
  
  func averageGrade() -> Double {
    
    guard let grades = grades?.allObjects as? [AssignmentGradeEntity] else { return 0 }
    guard grades.count > 0 else { return 0 }
    return grades.reduce(0, { $0 + $1.grade })/Double(grades.count)
  }
  
}

There are other ways to customize your Core Data class. Earlier, I mentioned using the "Editor-> Create NSManagedObject subclass" menu item. If you want to do this, you can select the Manual/None choice from the CodeGen drop down. If you want the build system to only manage the +CoreDataProperties extension, you can choose Category/Extension - but if you do that you’ll be responsible for creating the equivalent of the AssignmentEntity+CoreDataClass where the AssignmentEntity class was defined as a subclass of NSManagedObject.
