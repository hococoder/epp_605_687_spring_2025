Overview

Logging (Apple Documentation)
Generating Log Messages from Your Code (iOS 14)

While the use of a strategic "print" statement may provide some important information when trying to debug your app, sometimes you need a little bit more. Luckily, iOS 10 and above, macOS 10.12 and above, tvOS 10.0 and above, and watchOS 3.0 and above all support the "Unified Logging System". This system helps you get a handle on problems that can't be seen easily using other techniques, including:

  • When you can't attach the debugger - for example when you are diagnosing problems on a user's machine or device
  • When the problem is intermittent, and hard to catch in the debugger
  • When you want to get a more general sense of your app's behavior in everyday use (when certain tasks start or end, for example) and want to collect data over a broader range of time.

The logging system used here is very similar to the built in Logging library in Java. Let's look at what you need to know about making and formatting messages, and make some examples as we go.

Making Log Entries

To get started with logging in your class or struct, you need to import the os framework

import os.log

The main class you will be working with for logging is OSLog.

This has been updated in iOS 14, where the new `Logger` class is used in Swift, with a cleaner API.

If you've ever looked at the console on the device or your Mac, it has a LOT of information in it. Luckily, OSLog and Logger takes in 2 custom strings that can help differentiate your apps from others on the device.

  • subsystem can identify a set of functionality within your app. Threads or long running background processes are examples here. Use reverse DNS notation here, e.g. edu.jhu.ep.steele.josh.EPGradeBook.subsystem
  • category can identify a particular component or module in your app. You might define your user interface, databases code, and back end logic as separate categories. Any naming convention you would like to use is fine here.

Here are some examples of making log entries:

//iOS 13
import os.log

let logger = OSLog(subsystem: "edu.jhu.ep.steele.josh.EPGradeBook.serverMonitor", category: "networking")

os_log(OSLogType.debug, log: logger, "This is a debug message")

//iOS14 - the same import is used (import os.log)

let logger2 = Logger()

logger2.info("This is a debug message")

As you can see, the iOS 13 way doesn't have the typical Swift/Object Oriented way of doing things (this comes with Logger in the iOS 14 example), instead relying on old-style C/Objective-C system calls to log the message.

Logging Levels

Again, similar to the Java logging API, The Unified Logging System has its own set of levels. The following chart comes from here, which is centered around the new iOS 14 Logger framework, but the concepts for each level are the same.

Keep in mind that the the higher in severity you go, the more overhead you incur. Debug messages stay in memory, but Faults write to the log along with other information associated with the process.

Custom Data and Formatting

There is an extra argument in the os_log call that can take a series of arguments that allow you to customize your message string.

//iOS 13
os_log("The very important variable is %d", log: logger, type: OSLogType.debug, importantVariable)

Note that this uses a different method than the one we used earlier - the message comes first and is of type StaticString

in iOS 14 with `Logger`, the usual Swift string interpolation can be used instead of having to resort to C style formatting

//iOS 14
log.info("The name of the person is \(personName)")

Also, in iOS 14, you can define subsystems and categories when making your logger

let logger3 = Logger.init(subsystem: "com.josh.myApp", category: "databaseAccess")

The above subsystem will let you easily search in the Console app for "com.josh.myApp" and the category will quickly let you see that the log message comes from your database access code, as opposed to some other part of your app.
