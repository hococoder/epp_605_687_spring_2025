Week 1 - Structs and Classes

Previous: Generics

Next: Protocols and Extensions



Structs

Unlike Java, Swift has a dedicated Struct type. These are different in a few ways from Classes (see below):

  • Structs are passed by value
  • Structs get automatically generated initializers - a struct can be defined with a little as the properties that make it up!
  • Structs can adopt protocols, but can't inherit from classes.
  • Are considered "lightweight", and are the default (recommended type) when using Swift.

Classes

Classes in Swift are similar to what you're used to in languages like Java:

  • Classes are passed by reference
  • They do not get automatically created initializers, and need to have all values initialized before they can be used (even with Optionals)
  • They can use inheritance to borrow properties and methods from their parent
  • Can be used to get interoperability with Objective-C code (the language used by Apple for many years before Swift, which is still used some today!)

Structs vs Classes

As you've seen, Swift has a dedicated Struct type, in addition to a Class type. Unlike C based languages, you don't need to define an interface (.h) file - that gets generated for you behind the scenes. So, when you are implementing your code, which should you use, structs or classes? Let's break things down. According to the Swift Programming guide, both types can:

  • Define properties to store values
  • Define methods to provide functionality
  • Define subscripts to provide access to their values using subscript syntax
  • Define initializers to setup initial state
  • Be extended to expand functionality beyond a default implementation
  • Conform to protocols to provide standard functionality of a certain kind

Classes, however, have some extra features

  • Inheritance enables one class to inherit characteristics of another
  • Type casting enables you to check and interpret the type of a class instance at runtime
  • Deinitializers enable an instance of a class to free up any resources it has assigned
  • Reference counting allows more than one reference to a class instance

This extra behavior for classes comes at the cost of increased complexity. So, as a general rule, Swift prefers to have things defined as structs, which are passed by value throughout your code. Here's an example of structs and classes:

struct Person {
  var name: String
  let dob: Date
}

class Teacher {
  let person: Person
  var classesNamesTaught: [String]?
  
  init(person: Person) {
    self.person = person
  }
}

The Person structure doesn't has an initializer - we're using the struct's capability to build a member-wise initializer automatically for us. If you need to do other things at struct creation, you can make your own initializer. The Teacher class has an init function which sets the person property. The class also has a classesNamesTaught property, which is an optional array of Strings. That array may exist, or it may be nil; if it does exist, it will hold non-optional Strings (note that the optional is on the array, not the String. If it was an array of optional Strings, it would be [String?])

A great article on choosing between Structs and Classes from Apple can be found [here](https://developer.apple.com/documentation/swift/choosing_between_structures_and_classes)
