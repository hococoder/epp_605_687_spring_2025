Functions are vital in any language, and Swift has its own way of presenting a function, calling it, and referring to it. Let's take a look at a quick example.

func sayHello(name: String, year: Int) -> String {

  return "Hello there \(name), the year is \(year)"

}

sayHello(name: "Josh", year: 2021)

Function signatures

Let's look at the function signature first:

func sayHello(name: String, year: Int) -> String

The func keyword precedes the name of the function sayHello, which is followed by the parameter list in parentheses. The return type is listed after the parameters list (and a "->") as a String.

The parameters list looks flipped from what you might be used to in other languages: the parameter label is listed first, followed by the type (e.g. name: String). These parameter labels are part of the signature, as they are used when the function is called

sayHello(name: "Josh", year: 2021)

This way, when you are calling the function (or reviewing code that calls the function), if the label is properly descriptive, you can tell exactly what the parameter is that is being passed into the function. You can also add in argument labels to make the "as-called" function more descriptive for the caller:

func sayHello(to name: String, in year: Int) -> String{

  return "Hello there \(name), the year is \(year)"

}

sayHello(to: "Josh", in: 2021)

This naturally reads as "Say hello to Josh in 2021" - a very descriptive call!

Return types

A few notes on return types:

  • If there is no return type, you can either list it as -> Void or just omit the return type entirely
  • Single return values can be stated as above (e.g. -> String)
  • Multiple return values can be stated as a tuple (e.g. -> (String, Int, Double) would return a tuple containing a String, Int, and Double, in that order)

Opaque Return Types

A new keyword introduced in Swift 5.1 is the some keyword. This keyword allows for generic protocols (which either have associated types or references to Self) to be used as return types, along with concrete types such as non-generic protocols, classes or structs (all of which you'll learn about later). This keyword is seen all over the place in SwiftUI, and allows us to be very generic when defining our views - as long as they conform to the View protocol. This eliminates to need to specify concrete types or use type erasure to hide the type in a more generic box type before being returned.

Omitted Return keywords

Thanks to type inference, the compiler can, for a single expression function, allow for the return keyword to be omitted. Our example above can be updated to be:

func sayHello(name: String, year: Int) -> String {

  "Hello there \(name), the year is \(year)"

}

This isn't a big change, but it is used a lot (especially in closures, which you'll learn about in the next module)
