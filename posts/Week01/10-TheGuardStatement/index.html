The guard Statement

Guard statements take all the features of an if let and combine it with a bouncer at a bar. Not sure what I mean? let's have a look:

func calculate(valueA: Int, valueB:Int) -> Int?
{
  guard valueA < 11, valueB < 11 else { print("Get out of here!"); return nil }
  return valueA*valueB
}

This method takes in two Ints, and multiples them together. Unfortunately the method only knows its times tables up through 10, so we need to make sure no values greater than 10 come in. That's where the guard statement helps. If either valueA or valueB is greater than 10, we immediately exit the function, and return a nil value. Here, the guard statement acts like a bouncer - if it checks your ID, and you don't pass - you get bounced.

Here are a few facts about guard statements.

  • First, they must exit the current control at the end of their else statement. You can perform other steps before that if you wish, but the last statement must be something like return, break or continue.
  • Second, they can work on regular parameters in addition to optional ones. For optional checks, you can use guard let, which acts very similar to a if let, except that the value you put your optional into is scoped until the end of the function. For example, let's make our parameters optionals in our calculate method (now called calculate2):
func calculate2(valueA: Int?, valueB:Int?) -> Int?
{
  guard let valA = valueA, let valB = valueB else { print("You're nothin!"); return nil}
  guard valA < 11, valB < 11 else { print("Get out of here!"); return nil }
  return valA*valB
}

As you can see, first we performed a guard let to check the optionals to make sure they had values in them. If they didn't the first guard statement fails and the control flow leaves the method. If they do have values however, those values are checked to make sure they are 10 or less, and only if that is true does the last line, where they get multiplied together, get executed. Guard statements can come in handy when validating a form where a required set of data needs to be present in order to continue.

A clear advantage of a guard statement is avoiding the dreaded "Pyramid of Doom". Have you ever written a series of nested if loops that check for a series of conditions, one after the other, your indentation moving more and more to the right as you add more and more checks? Let's see what the body of our method would look like this way:

func calculate3(valueA: Int?, valueB: Int?) -> Int?
{
  if let valA = valueA, let valB = valueB
  {
    if valA < 11, valB < 11
    {
      return valA*valB
    }
    else
    {
      print("Get out of here!")
      return nil
    }
  }
  else
  {
    print("You're nothin!")
    return nil
  }
}

We went from 3 to 17 lines! Granted, we could play with the formatting to get things down a bit, but if we had more checks that indentation would just keep growing the pyramid of doom closer and closer to the right side of the screen.
